# AGENTS.md

# Project

Re:Mine은 사용자가 저장해두고 잊어버린 콘텐츠를 다시 발견하고, 학습과 실제 성과로 연결할 수 있도록 돕는 모바일 중심 웹 애플리케이션입니다.

서비스의 핵심 흐름은 다음과 같습니다.

> 摘 — 잊힌 배움을 다시 찾아내고
>
> 積 — 꾸준히 지식을 쌓으며
>
> 績 — 배움을 실제 성과로 연결한다

Re:Mine은 단순한 북마크 저장 서비스가 아닙니다.

사용자가 YouTube, GitHub, 기술 블로그, 논문, 디자인 레퍼런스 등의 링크를 저장하고, 다시 확인하고, 학습 기록과 결과물로 발전시킬 수 있도록 돕는 서비스입니다.

# Tech Stack

## Frontend

- React
- Vite
- TypeScript
- Tailwind CSS
- React Router
- Zustand
- PWA
- Service Worker
- Web Push API
- Web Share Target API

프로젝트에 설치되지 않은 라이브러리는 임의로 추가하지 않습니다.

새 라이브러리가 필요하다면 기존 코드와 현재 의존성만으로 구현할 수 있는지 먼저 확인합니다.

## Backend

- 백엔드는 다른 팀에서 개발하고 관리합니다.
- 명시적으로 요청받지 않는 한 백엔드 코드를 생성하지 않습니다.
- 백엔드와는 REST API를 통해 통신한다고 가정합니다.
- API 명세에 없는 엔드포인트나 데이터 구조를 임의로 만들지 않습니다.

# Core Principles

- 모바일 환경을 우선으로 구현합니다.
- 기존 코드를 먼저 확인하고 재사용합니다.
- UI, 비즈니스 로직, 서버 통신, 전역 상태를 분리합니다.
- 컴포넌트 하나에 지나치게 많은 책임을 넣지 않습니다.
- 가장 단순하고 유지보수하기 쉬운 구현을 선택합니다.
- 요청받지 않은 구조 변경이나 대규모 리팩터링을 하지 않습니다.
- 기능 구현 범위를 벗어난 파일을 임의로 수정하지 않습니다.

# Architecture

- React 함수형 컴포넌트를 사용합니다.
- 라우팅은 React Router를 사용합니다.
- 페이지 컴포넌트는 화면 구성에 집중합니다.
- 비즈니스 로직은 커스텀 훅이나 유틸리티로 분리합니다.
- API 요청 로직은 UI 컴포넌트에서 분리합니다.
- 여러 화면에서 공유하는 클라이언트 상태는 Zustand로 관리합니다.
- 컴포넌트 내부에서만 사용하는 상태는 `useState`를 사용합니다.
- 파생 가능한 값은 별도의 상태로 저장하지 않습니다.
- 불필요한 Context API 사용을 피합니다.
- 무조건적인 추상화보다 현재 요구사항에 맞는 단순한 구조를 우선합니다.

# Folder Rules

## `app`

애플리케이션의 초기 설정과 전역 구성을 관리합니다.

예시:

- 라우터 설정
- 전역 Provider
- 앱 최상위 컴포넌트
- 전역 에러 처리
- 전역 초기화 로직

## `pages`

라우트 단위 화면 컴포넌트를 관리합니다.

페이지 파일은 가능한 한 작게 유지합니다.

페이지 컴포넌트의 주요 역할은 다음과 같습니다.

- 페이지 레이아웃 구성
- feature 컴포넌트 조합
- URL 파라미터 및 쿼리 파라미터 전달
- 페이지 수준의 로딩 및 에러 상태 처리

페이지 파일에 큰 UI나 복잡한 비즈니스 로직을 직접 작성하지 않습니다.

## `features`

서비스 도메인별 코드를 관리합니다.

예시:

- 링크 저장
- 링크 공유받기
- 컬렉션
- 복습
- 성과 기록
- 알림
- 사용자

기능에 종속된 컴포넌트, API, 훅, 타입, 유틸리티는 해당 feature 내부에 둡니다.

## `components/ui`

서비스 도메인과 무관한 재사용 가능한 기본 UI를 관리합니다.

예시:

- Button
- Input
- Textarea
- Modal
- BottomSheet
- Badge
- Spinner
- EmptyState

## `components/common`

여러 기능에서 공유하지만 서비스의 맥락을 포함하는 컴포넌트를 관리합니다.

예시:

- LinkCard
- ContentThumbnail
- CategoryChip
- ErrorMessage

## `components/layout`

레이아웃과 내비게이션 관련 컴포넌트를 관리합니다.

예시:

- AppLayout
- Header
- BottomNavigation
- PageContainer

## `hooks`

여러 기능에서 공통으로 사용하는 커스텀 훅을 관리합니다.

기능 전용 훅은 해당 feature 내부에 둡니다.

## `services`

외부 서비스와의 통신 및 공통 통신 설정을 관리합니다.

예시:

- API 클라이언트
- 인증 토큰 처리
- Push API
- Service Worker 등록
- 브라우저 스토리지 접근

## `store`

여러 화면 또는 관련 없는 컴포넌트 사이에서 공유되는 클라이언트 상태를 관리합니다.

## `utils`

도메인에 종속되지 않은 순수 유틸리티 함수를 관리합니다.

## `types`

프로젝트 전체에서 공유되는 공통 타입을 관리합니다.

기능 전용 타입은 해당 feature 내부에 유지합니다.

## `constants`

프로젝트 전체에서 사용하는 상수를 관리합니다.

# Existing Code First

새로운 파일이나 기능을 만들기 전에 반드시 기존 프로젝트를 먼저 확인합니다.

1. 관련 파일을 검색합니다.
2. 기존 컴포넌트를 확인합니다.
3. 기존 훅을 확인합니다.
4. 기존 유틸리티를 확인합니다.
5. 기존 API 함수와 타입을 확인합니다.
6. 기존 Zustand store를 확인합니다.
7. 기존 디자인 패턴과 네이밍을 확인합니다.
8. 기존 구현을 확장할 수 있는지 판단합니다.

기존 구현으로 해결할 수 있다면 새 구현을 만들지 않습니다.

다음과 같은 중복을 만들지 않습니다.

- 동일한 역할의 UI 컴포넌트
- 동일한 API 요청 함수
- 동일한 타입
- 동일한 URL 파싱 함수
- 동일한 날짜 포맷 함수
- 동일한 Zustand 상태
- 동일한 모달 또는 바텀시트
- 동일한 링크 카드 UI

# Development Workflow

새로운 기능을 구현할 때 다음 순서를 따릅니다.

1. 사용자의 요구사항을 확인합니다.
2. 관련 기존 코드를 검색합니다.
3. 현재 프로젝트 구조와 패턴을 파악합니다.
4. 새로운 코드가 실제로 필요한지 판단합니다.
5. 구현 범위를 최소화합니다.
6. 타입을 먼저 확인하거나 정의합니다.
7. UI와 로직을 분리해 구현합니다.
8. 로딩, 에러, 빈 상태를 확인합니다.
9. 모바일 화면과 접근성을 확인합니다.
10. TypeScript와 ESLint 오류를 확인합니다.
11. 사용하지 않는 코드와 import를 제거합니다.

기존 프로젝트가 동작하고 있다면 요청받지 않은 부분을 함께 수정하지 않습니다.

# Components

- 컴포넌트는 한 가지 주요 책임을 가집니다.
- 큰 컴포넌트보다 작은 컴포넌트의 조합을 선호합니다.
- 재사용 가능성이 분명한 UI만 공통 컴포넌트로 분리합니다.
- 한 번만 사용하는 작은 UI를 무조건 별도 파일로 분리하지 않습니다.
- 컴포넌트가 약 200줄을 넘거나 이해하기 어려워지면 분리를 검토합니다.
- Props drilling이 심해지기 전까지 불필요한 전역 상태를 만들지 않습니다.
- JSX 안에 복잡한 연산이나 긴 조건문을 작성하지 않습니다.
- 이벤트 핸들러는 의미 있는 이름을 사용합니다.
- 컴포넌트 안에서 API 요청 URL을 직접 작성하지 않습니다.

권장 컴포넌트 내부 순서는 다음과 같습니다.

1. Imports
2. Types
3. Constants
4. Component
5. State
6. Custom Hooks
7. Derived Values
8. Event Handlers
9. Helper Functions
10. JSX

# Routing

- 라우팅은 React Router를 사용합니다.
- 라우트 정의는 한 곳에서 관리합니다.
- URL 문자열을 여러 컴포넌트에 반복해서 하드코딩하지 않습니다.
- 반복해서 사용하는 경로는 상수 또는 경로 생성 함수로 관리합니다.
- 존재하지 않는 경로를 위한 Not Found 페이지를 제공합니다.
- 동적 경로 파라미터는 반드시 타입과 유효성을 확인합니다.
- 쿼리 파라미터를 신뢰하지 말고 파싱과 검증을 거칩니다.

예시:

```tsx
export const ROUTES = {
  home: "/",
  links: "/links",
  linkCreate: "/links/new",
  linkDetail: (linkId: string) => `/links/${linkId}`,
  share: "/share",
  review: "/review",
} as const;
```

# State Management

상태의 범위에 맞는 도구를 선택합니다.

## `useState`

하나의 컴포넌트 또는 가까운 하위 컴포넌트에서만 사용하는 상태에 사용합니다.

예시:

- 입력값
- 드롭다운 열림 상태
- 아코디언 상태
- 체크박스 상태
- 컴포넌트 로컬 로딩 상태
- 임시 선택 상태

## `useReducer`

서로 연관된 로컬 상태의 전이가 복잡할 때 사용합니다.

예시:

- 여러 단계로 이루어진 링크 등록 폼
- 상태 전이가 명확한 온보딩
- 여러 액션을 가지는 복잡한 필터 UI

단순한 상태에 불필요하게 `useReducer`를 사용하지 않습니다.

## Zustand

여러 화면 또는 서로 관련 없는 컴포넌트가 공유해야 하는 클라이언트 상태에 사용합니다.

예시:

- 인증 사용자 정보
- 온보딩 진행 상태
- 전역 모달 또는 바텀시트
- 선택된 필터
- 알림 권한 상태
- 임시 공유 링크 데이터
- 테마
- 앱 전역 UI 상태

규칙:

- Store 파일은 `src/store`에 둡니다.
- Store는 기능 또는 도메인 단위로 분리합니다.
- 하나의 거대한 전역 Store를 만들지 않습니다.
- 서버에서 가져온 모든 데이터를 무조건 Zustand에 저장하지 않습니다.
- 한 컴포넌트만 사용하는 상태는 `useState`를 우선합니다.
- 파생 값은 Store에 중복 저장하지 않습니다.
- Store action의 이름은 상태 변경 목적이 드러나게 작성합니다.

# API Integration

- API 명세에 없는 엔드포인트를 만들지 않습니다.
- 요청 및 응답 데이터 구조를 임의로 가정하지 않습니다.
- API 명세와 백엔드 응답 타입을 기준으로 구현합니다.
- API URL을 컴포넌트에 직접 작성하지 않습니다.
- 공통 Base URL은 환경 변수 또는 API 클라이언트에서 관리합니다.
- 기능별 API 요청은 `features/{feature}/api`에 둡니다.
- 공통 API 클라이언트는 `services` 또는 `lib`에 둡니다.
- API 응답 타입을 명시합니다.
- HTTP 에러와 네트워크 에러를 구분할 수 있도록 처리합니다.
- 로딩, 성공, 실패 상태를 화면에서 적절하게 표현합니다.
- 사용자에게 내부 서버 에러 메시지를 그대로 노출하지 않습니다.
- 요청 취소나 컴포넌트 언마운트 상황을 고려합니다.

API가 준비되지 않았다면 임시 Mock 데이터를 사용할 수 있습니다.

이 경우 반드시 다음을 지킵니다.

- 실제 API처럼 보이는 가짜 엔드포인트를 만들지 않습니다.
- Mock임을 분명히 표시합니다.
- 실제 연동 시 제거할 수 있도록 TODO를 남깁니다.
- UI 컴포넌트 내부에 큰 Mock 데이터를 직접 작성하지 않습니다.

예시:

```tsx
//TODO: Replace with the actual API response after backend integration.
export type LinkResponse = {
  id: string;
  title: string;
  url: string;
};
```

# Environment Variables

- 환경 변수는 `import.meta.env`를 통해 접근합니다.
- 클라이언트에서 사용하는 환경 변수는 `VITE_` 접두사를 사용합니다.
- 비밀 키나 서버 전용 키를 프론트엔드 환경 변수에 저장하지 않습니다.
- 환경 변수 값에 대한 타입을 정의합니다.
- 환경 변수 누락 시 가능한 한 명확한 에러를 제공합니다.
- `.env` 파일을 Git에 커밋하지 않습니다.
- `.env.example`에는 필요한 키 이름만 작성합니다.

예시:

```tsx
const apiBaseUrl = import.meta.env.VITE_API_BASE_URL;
```

# PWA

Re:Mine은 모바일에서 앱처럼 사용할 수 있는 PWA를 목표로 합니다.

PWA 관련 코드는 일반 UI 코드와 분리합니다.

## Manifest

- 앱 이름, 아이콘, 시작 경로, 배경색, 테마 색상을 명시합니다.
- `display`는 서비스 요구사항에 따라 `standalone`을 우선 검토합니다.
- 모든 아이콘 경로가 실제 파일과 일치하는지 확인합니다.
- Manifest 설정 변경 시 설치된 앱을 재설치해야 할 수 있음을 고려합니다.

## Service Worker

- Service Worker 로직은 일반 React 컴포넌트에 작성하지 않습니다.
- Service Worker 등록 코드는 별도 서비스 또는 초기화 파일로 분리합니다.
- Service Worker는 DOM에 직접 접근할 수 없다는 점을 고려합니다.
- 캐시 정책을 명확하게 구분합니다.
- 오래된 캐시가 무한히 남지 않도록 버전 관리와 정리 로직을 고려합니다.
- 개발 환경과 프로덕션 환경의 동작 차이를 고려합니다.
- Service Worker 업데이트 시 사용자에게 오래된 앱 상태가 유지되지 않도록 주의합니다.

## Offline

- 모든 API 기능이 오프라인에서 동작한다고 가정하지 않습니다.
- 오프라인에서 제공할 수 있는 기능과 제공할 수 없는 기능을 구분합니다.
- 네트워크 연결이 필요한 기능에는 적절한 안내를 제공합니다.
- 저장되지 않은 사용자 입력이 네트워크 오류로 유실되지 않도록 고려합니다.

# Share Target

Re:Mine은 Android PWA에서 다른 앱이 공유한 URL을 전달받을 수 있습니다.

예상 흐름:

```
외부 앱에서 공유하기
→ 공유 대상에서 Re:Mine 선택
→ Re:Mine 링크 등록 화면 실행
→ 공유받은 URL이 입력된 상태로 표시
→ 사용자가 확인 후 저장
```

규칙:

- 공유 데이터는 신뢰할 수 없는 외부 입력으로 취급합니다.
- `title`, `text`, `url` 값이 항상 모두 전달된다고 가정하지 않습니다.
- 공유 앱에 따라 URL이 `url`이 아니라 `text`에 포함될 수 있음을 고려합니다.
- URL 추출 로직은 하나의 공통 함수로 관리합니다.
- URL의 형식과 프로토콜을 검증합니다.
- `http`와 `https` 이외의 위험한 프로토콜을 허용하지 않습니다.
- 쿼리 파라미터를 그대로 HTML로 렌더링하지 않습니다.
- 공유 직후 자동 저장보다 사용자가 확인한 뒤 저장하는 흐름을 우선합니다.
- 로그인되지 않은 경우 공유 데이터를 잃지 않도록 임시 보관 흐름을 고려합니다.
- 공유 링크 중복 여부는 서버 정책과 함께 처리합니다.

예시:

```tsx
type SharedContent = {
  title?: string;
  text?: string;
  url?: string;
};
```

URL 추출과 검증은 UI 컴포넌트 내부에 반복해서 작성하지 않습니다.

# Push Notifications

- 사용자 동의 없이 알림 권한을 요청하지 않습니다.
- 페이지 진입 즉시 알림 권한 팝업을 띄우지 않습니다.
- 알림 기능의 목적을 설명한 뒤 사용자 행동을 통해 권한을 요청합니다.
- 알림 권한 상태는 `default`, `granted`, `denied`로 구분합니다.
- 거부된 권한을 반복해서 요청하지 않습니다.
- Push Subscription 정보는 백엔드 API를 통해 등록 및 해제합니다.
- Subscription 데이터를 로그에 불필요하게 출력하지 않습니다.
- 알림 클릭 시 이동할 경로는 앱 내부 경로인지 검증합니다.
- 외부 URL을 알림 데이터에서 그대로 열지 않습니다.
- 앱이 이미 열려 있다면 기존 창을 포커스하거나 해당 경로로 이동하는 흐름을 우선합니다.
- 앱이 열려 있지 않다면 새로운 창을 엽니다.
- 알림 타입에 따라 이동 경로를 명확하게 관리합니다.

예시:

```tsx
export type NotificationType =
  "LINK_REMINDER" | "REVIEW_READY" | "WEEKLY_REPORT";
```

# Imports

가능한 한 절대 경로 import를 사용합니다.

Vite alias는 `@`가 `src`를 가리키도록 설정합니다.

좋은 예:

```tsx
import { Button } from "@/components/ui/Button";
import { LinkCard } from "@/features/link/components/LinkCard";
```

피해야 할 예:

```tsx
import { Button } from "../../../../components/ui/Button";
```

같은 디렉터리의 가까운 파일은 상대 경로를 사용할 수 있습니다.

예시:

```tsx
import { LinkFormField } from "./LinkFormField";
```

# Accessibility

- 아이콘만 있는 버튼에는 `aria-label`을 제공합니다.
- 모든 입력 요소에는 연결된 `label`을 제공합니다.
- 클릭 가능한 `div` 대신 `button`이나 `a`를 사용합니다.
- 키보드로 모든 주요 기능을 사용할 수 있어야 합니다.
- 포커스 표시를 제거하지 않습니다.
- 모달이 열리면 포커스 이동과 복귀를 고려합니다.
- 충분한 색상 대비를 유지합니다.
- 색상만으로 상태를 구분하지 않습니다.
- 이미지에는 적절한 `alt`를 제공합니다.
- 장식용 이미지는 빈 `alt`를 사용합니다.
- 로딩 상태는 스크린 리더가 이해할 수 있도록 처리합니다.
- 애니메이션은 사용자의 reduced motion 설정을 존중합니다.

# UI Principles

- 모바일 중심으로 설계합니다.
- 작은 화면에서도 주요 기능을 쉽게 사용할 수 있어야 합니다.
- 사용자가 링크를 빠르게 저장할 수 있도록 입력 단계를 최소화합니다.
- 사용자가 현재 어떤 상태인지 명확하게 보여줍니다.
- 저장, 실패, 중복, 오프라인 상태에 대한 피드백을 제공합니다.
- 장식보다 정보의 가독성과 행동의 명확성을 우선합니다.
- 동일한 기능은 동일한 UI 패턴을 사용합니다.
- 버튼, 간격, 색상, 타이포그래피를 일관되게 유지합니다.
- 빈 화면에는 다음 행동을 알 수 있는 안내를 제공합니다.
- 오류 메시지는 문제와 가능한 해결 방법을 함께 전달합니다.

# Performance

- 불필요한 리렌더링을 만들지 않습니다.
- 모든 값에 습관적으로 `useMemo`나 `useCallback`을 사용하지 않습니다.
- 실제 성능 문제나 참조 안정성이 필요한 경우에만 메모이제이션합니다.

- 목록의 key에 배열 index를 사용하지 않습니다.
- 긴 목록이 성능 문제를 일으키면 페이지네이션 또는 가상화를 검토합니다.
- 라우트 단위 코드 스플리팅을 고려합니다.
- 초기 로딩에 필요하지 않은 기능은 지연 로딩을 검토합니다.
- Service Worker 캐시가 최신 배포를 방해하지 않도록 관리합니다.

- 앱 전체가 중단되지 않도록 페이지 또는 기능 단위 에러 처리를 고려합니다.

# Code Style

- 기존 코드 스타일과 포맷을 유지합니다.
- 요청받지 않은 전체 파일 포맷 변경을 하지 않습니다.

# Git and Scope

- 요청과 관계없는 파일을 수정하지 않습니다.
- 자동 생성 파일을 임의로 직접 수정하지 않습니다.
- 기존 환경 설정을 확인하지 않고 덮어쓰지 않습니다.
- `.env` 파일이나 민감한 정보를 커밋하지 않습니다.
- 패키지 버전을 이유 없이 변경하지 않습니다.
- `package-lock.json`은 실제 의존성 변경이 있을 때만 수정합니다.
- 기능 구현과 무관한 대규모 포맷 변경을 하지 않습니다.
- 삭제가 필요한 파일은 사용 여부를 먼저 확인합니다.

# AI Instructions

코드를 생성하거나 수정할 때 반드시 다음을 따릅니다.

- 작업 전에 관련 기존 코드를 검색합니다.
- 프로젝트의 현재 폴더 구조를 우선합니다.
- 기존 컴포넌트, 훅, 유틸리티, 타입, API 함수, Store를 재사용합니다.
- 동일한 기능을 중복 구현하지 않습니다.
- 요청 범위 밖의 파일을 수정하지 않습니다.
- 사용자가 요청하지 않은 라이브러리를 설치하지 않습니다.
- API 엔드포인트와 응답 형식을 임의로 만들지 않습니다.
- 백엔드 코드를 임의로 생성하지 않습니다.
- 타입 안정성을 유지합니다.
- `any`, `@ts-ignore`, 무리한 타입 단언으로 오류를 숨기지 않습니다.
- TypeScript와 ESLint 오류의 원인을 해결합니다.
- 새로운 파일은 실제로 유지보수에 도움이 되는 경우에만 만듭니다.
- 새로운 파일을 만든다면 그 이유가 명확해야 합니다.
- 가장 단순하고 유지보수 가능한 해결책을 우선합니다.
- 기존 디자인과 아키텍처를 보존합니다.
- 임시 구현에는 TODO와 교체 조건을 명확히 남깁니다.
- 보안에 영향을 주는 외부 입력, URL, 권한, 토큰을 신뢰하지 않습니다.
- 작업이 끝난 후 변경된 파일과 핵심 변경 사항을 설명합니다.
- 실행하지 않은 테스트나 확인하지 못한 동작을 실행했다고 말하지 않습니다.
